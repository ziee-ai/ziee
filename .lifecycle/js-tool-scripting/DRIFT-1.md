# DRIFT-1 — implementation vs plan (round 1)

Audit of the implemented backend against PLAN.md / DECISIONS.md.

- **DRIFT-1.1** — verdict: impl-wins — DEC-1/ITEM-1 said do NOT enable rquickjs's `parallel` feature. Implementation MUST enable it: rquickjs gates the `Send`/`Sync` impls for `AsyncRuntime`/`AsyncContext` behind `#[cfg(feature = "parallel")]` (verified in `rquickjs-core-0.12.1/src/context/async.rs:246`). Without it the runtime is `!Send`, so awaiting the evaluation inside the `tokio::spawn`ed + `#[async_trait]`-`Send`-boxed chat-turn task fails to compile. We drive ONE runtime from a SINGLE task and never touch it concurrently across threads, so the experimental concurrent-access caveat does not apply — `parallel` is used only for the `Send` marker so the future may migrate between worker threads between awaits. DECISIONS.md DEC-1 amended with this rationale; the `Cargo.toml` comment documents it at the source. Also required `runtime.rs`'s capture sinks to become `Send` (`Rc`/`Cell` → `Arc<Mutex>`/atomics) — behaviourally identical (single-threaded, no contention); the 9 runtime unit tests still pass.
- **DRIFT-1.2** — verdict: resolved — the `runtime::evaluate` inject callback was planned as `FnOnce(&Ctx, &Object)`; implemented as `FnOnce(&Ctx)` (the `Object` param was unused — `host_bridge::install` evals a prelude against the already-set `globalThis.ziee` global). Collapsing to a single lifetime also fixed a `for<'js>` HRTB unification error. Pure internal signature detail; no plan-item behaviour changes.
- **DRIFT-1.3** — verdict: impl-wins — ITEM-8 planned the run_js execution intercept "at mcp.rs ~2311". Implemented as a first branch of the existing inline-intercept `if` chain (the same site that intercepts `ask_user`), plus a new `execute_run_js_call` helper method that gathers the accessible tool set via `list_tools`. Same site, more precise than the plan's line estimate; matches the `ask_user` precedent exactly. PLAN wording is descriptive, not normative here — no amendment needed.
- **DRIFT-1.4** — verdict: none — the auto-approved allowlist was planned as passed-in; implemented by flattening `auto_approved_servers` + `user_auto_approved` (via `AutoApprovedServer::contains_tool`) into a `HashSet<(Uuid,String)>` inside the intercept helper, exactly matching the normal loop's classification. Conforms to plan.

- **DRIFT-1.5** — verdict: impl-wins — PLAN's Files-to-touch listed `src-app/desktop/ui/**` mirrors for the frontend (ITEM-13/14). Reality: `desktop/ui` has NO `mcp` module (its modules are app/auth/desktop-base/file-dialog/hardware/host-mount/layouts/llm-provider/memory/remote-access/settings/tunnel-auth/updater/window) — the desktop shell doesn't render chat/MCP tool approvals. So the frontend applies ONLY to `src-app/ui`; desktop/ui received only the regenerated `openapi.json` + `api-client/types.ts` (excluded from the UI-touch gate). No desktop component mirror was created or needed. PLAN over-specified; no product impact.
- **DRIFT-1.6** — verdict: resolved — adding the `run_js_approval` testids + the `JsToolApprovalContent` surface required regenerating `testIds.generated.ts` and adding a `{kind:'via'}` allowlist entry in `dev/gallery/coverage.ts` (+ regenerating `galleryCoverage.generated.ts`/`stateMatrix.generated.ts`) — expected mechanical consequences of a new content-renderer, reconciled so `npm run check` is green.
