# HUMAN_FEEDBACK — collapse-border-overlay

Human input during this feature, verbatim, with how each was resolved. The user
was consulted four times, each as an explicit option picker on a genuine
product/scope choice rather than an implementation question.

- **FB-1** [status: resolved] — "Plan now, reproduce first thing after approval" (choosing not to run the live gallery reproduction during planning, since it required `npm install` in a fresh worktree) → The plan was finalized on static analysis with the mechanism flagged as high-confidence-but-unverified, and the reproduction was made the first post-approval step, gating the fix. That gating mattered: the reproduction overturned the plan's stated root cause. [generalizable: yes — when a plan rests on an unverified causal claim, say so explicitly in the plan and make the verification a gate on the fix, rather than letting an unverified mechanism silently become the design premise]
- **FB-2** [status: resolved] — "Split at the last structural block (Recommended)" (choosing how the fix should handle interleaved `text → tool → text` turns) → Implemented as approved. Later reversed with the user's agreement (FB-4) after the blind audit proved it caused HIGH regressions. Recorded here as an approved decision that was correctly escalated rather than silently overridden.
- **FB-3** [status: resolved] — "Ship focused; document the pre-existing red" (on `npm run check` failing at `check:testid-registry` on the base commit itself) → The diff stays at 6 files; the pre-existing failure is documented in TEST_RESULTS.md with the verification transcript showing it reproduces on a pristine `origin/khoi`. Half the drift lives in the `sdk` submodule and is unfixable from a PR into `khoi`. [generalizable: yes — before claiming a gate is broken by your change, re-run it on a pristine base checkout; and never absorb an unrelated pre-existing cleanup into a bugfix diff just to turn a gate green]
- **FB-4** [status: resolved] — "Drop the split; ship the inset alone (Recommended)" (after the audit found the approved split caused two HIGH regressions) → `ChatMessage.tsx`, `collapsible.ts` and `collapsible.test.ts` reverted to base; only the `CollapsibleBlock` inset ships. [generalizable: yes — when an audit invalidates a design the human explicitly approved, bring them the evidence and the options rather than either silently reverting or shipping the known regression]

## Self-identified process failures (no human prompt; recorded because they are the transferable lessons)

- **FB-5** [status: resolved] — I tuned a test around a defect instead of recognising it. When the fixture's prose measured 368px and the message silently stopped collapsing, I recorded it as a fixture-sizing problem and enlarged the fixture from 8 to 12 copies. That WAS the HIGH regression surfacing — the clamp measuring only the trailing prose — and a blind auditor identified it from my own comment. → The defect is gone with the split; TEST-5 now pins height-bounding explicitly. [generalizable: yes — a surprising measurement while building a fixture is evidence about the CODE until proven otherwise; never resize a fixture to make a symptom go away without first asking what the symptom is telling you]
- **FB-6** [status: resolved] — I let a measurement I had already taken fall out of view. My very first probe recorded `thinking-card topRing = 0`, i.e. the top edge was clipped too. The split then moved cards out of the clamp entirely, masking it, and when the split was reverted I did not carry that measurement forward — the round-2 auditor had to re-find it. → Inset widened to all four sides and pinned by `expectRingRoom`. [generalizable: yes — when an approach changes, re-check the ORIGINAL measurements against the new approach; evidence collected under a discarded design does not automatically transfer]
- **FB-7** [status: resolved] — My first two attempts at the paint assertion were both wrong (a spurious scroll offset, then a window that included the card's interior and so passed a `ring-0` regression in dark mode). Only running each assertion against deliberately-broken code exposed them. → Every assertion in the spec is now validated against a broken build, not just a green one. [generalizable: yes — a new test assertion is not trustworthy until it has been observed to FAIL on the defect it claims to catch; run it against broken code before believing a green result]
