# PLAN_AUDIT — pdf-viewer-architecture

Audited each ITEM against the real codebase (handlers, routes, permissions,
viewer module, gallery mockApi, deps, CSP). Evidence inline.

## Breakage risk

- **No existing caller is broken.** `get_raw` is a *new* route; `download`,
  `preview`, `thumbnail`, `text` are untouched. Registering `/files/{file_id}/raw`
  BEFORE `/files/{file_id}` (like `/preview`, `/thumbnail`) avoids the
  catch-all-conflict the routes.rs comment warns about.
- The office/image path (`body.tsx` + `header.tsx` + the Document module entry)
  is **kept verbatim** — DOCX/RTF/ODT rendering is unaffected. Only the
  `application/pdf` entry is repointed to `PdfJsBody`.
- The backend ingest pipeline (`processing/pdf.rs`, `PREVIEW_PAGE_CAP`) is **not
  touched** (see DEC-2): real PDFs keep producing preview images at ingest
  (now unused by the new viewer, harmless) so nothing that reads
  `preview_page_count`/`has_thumbnail`/thumbnails for a PDF regresses (file
  cards, `File.store` auto-load gate at `File.store.ts:348/755`, truncation
  detection). The new viewer simply ignores those for PDFs.
- **Bundle size**: `pdfjs-dist` is large, but loaded via dynamic `import()` in
  `pdfjs.ts`, so it is a lazy chunk fetched only when a PDF viewer opens — the
  main bundle is unaffected. Confirmed no CSP blocks the worker/blob:
  `tauri.conf.json` `csp: null`; no `Content-Security-Policy` set server-side.

## Pattern conformance

- **ITEM-1/2** mirror `get_preview`/`get_preview_docs` (`handlers/management.rs:59,229`)
  and `download_file` (`handlers/download.rs`, `storage.load_original(user_id,
  blob_version_id, &ext)`): same `RequirePermissions<(FilesPreview,)>`
  (`permissions.rs:37`), same `get_by_id_and_user → not_found("File")` ownership
  check, same `[ (CONTENT_TYPE,…), (CONTENT_LENGTH,…), (CACHE_CONTROL,
  FILE_CONTENT_CACHE_CONTROL) ]` header array, same aide `*_docs` with
  `.response::<200, Json<BlobType>>()` (`types.rs:105`) → codegen emits a Blob
  method. `handlers/mod.rs` already `pub use management::*`, so no new export
  wiring. **Difference from `download`**: `FilesPreview` (not `FilesDownload`)
  gate + `Content-Disposition: inline`.
- **ITEM-5** mirrors `body.tsx`'s IntersectionObserver-on-the-OverlayScrollbars-
  viewport + reserved-height windowing.
- **ITEM-10** mirrors the existing `module.tsx` `FileViewerModule[]` two-entry
  split.
- **ITEM-11** mirrors the `overlay-file-preview-drawer` gallery entry
  (`overlays.tsx:420`) + the `mockApi` cassette/route registry.

## Migration collisions

- **None.** This feature adds NO SQL migration (no schema change — the raw-bytes
  endpoint reads existing blobs). `ls migrations/` is irrelevant here.

## OpenAPI regen

- **Required (ITEM-2).** The new route adds an operation → `openapi.json` +
  `api-client/types.ts` must be regenerated for **both** binaries via
  `just openapi-regen`, and `npm run check` (types parity golden test on the
  backend side; `tsc` on the frontend side) must pass in both workspaces. The
  regenerated `openapi.json`/`types.ts` are mechanically-generated and excluded
  from the phase-3/8 UI gate + the phase-6 coverage law (per skill note).

## Per-item verdicts

- **ITEM-1** — verdict: PASS — direct mirror of `get_preview`+`download_file`; perm `FilesPreview` exists; `load_original` signature confirmed; only a thin new handler.
- **ITEM-2** — verdict: CONCERN — needs `just openapi-regen` in BOTH binaries + `npm run check` parity in both workspaces; route must be placed before `/files/{file_id}`. Mechanical but must not be skipped.
- **ITEM-3** — verdict: CONCERN — `pdfjs-dist` is ABSENT from `node_modules` and both `package.json`s; install must update both workspaces + root lock + stay syncpack-clean. Worktree `node_modules` is symlinked to the main repo's, so the install strategy (per-workspace `-w` add, keep versions equal) is called out in DEC-4; verify `npx syncpack lint` clean.
- **ITEM-4** — verdict: PASS — `ApiClient.File.getRaw` (generated by ITEM-2) returns a Blob → `await blob.arrayBuffer()`; note (DEC-5) pdfjs `getDocument({data})` detaches the buffer, so pass a fresh `Uint8Array` copy; hook must destroy the loadingTask/document + abort fetch on unmount.
- **ITEM-5** — verdict: PASS — canvas-per-page + text layer is standard pdfjs usage; windowing pattern already proven in `body.tsx`.
- **ITEM-6** — verdict: PASS — page↔scroll mapping is local view state; pure helper is unit-testable.
- **ITEM-7** — verdict: PASS — fit-width/fit-page scale = viewport / page-viewport dims; pure helper is unit-testable; re-render on scale change.
- **ITEM-8** — verdict: PASS — pdfjs `getTextContent()` per page + match enumeration; highlight via text-layer spans. Match-ordering/cycling is a pure helper. Note: cross-span matches (a query spanning two text items) are a known pdfjs edge — helper normalizes per-page joined text with an index map.
- **ITEM-9** — verdict: PASS — native selection works once a correctly-positioned text layer exists (ITEM-5); no extra API.
- **ITEM-10** — verdict: PASS — module split is mechanical; office entry unchanged.
- **ITEM-11** — verdict: CONCERN — gallery `mockApi.ts` currently returns ONLY `jsonResponse`; a binary-response path must be added so `/files/{id}/raw` serves real PDF bytes for the `loaded` state (otherwise only the `error` state is reachable offline). Small, self-contained addition; also generally useful.

No `BLOCKED` verdicts. The three CONCERNs (regen, dep install, mockApi binary
support) are all resolved in-plan and tracked as decisions.
