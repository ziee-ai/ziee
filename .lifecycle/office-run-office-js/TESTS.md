# TESTS — office-run-office-js

Bipartite: every ITEM has ≥1 TEST; every TEST names valid ITEM(s), a tier, a
target file, and what it asserts. Tiers mirror the module's existing pattern —
in-source `#[cfg(test)]` (unit), `tests/office_bridge/` (integration), node
`--test` for the pane's pure helpers, and the `#[cfg(target_os="macos")]
#[ignore]` live op. **Not frontend-workspace work** (`taskpane.js` is under
`desktop/tauri/resources`, the generated desktop `openapi.json`/`types.ts` are
excluded) — so no `tier: e2e` spec is required by the gate.

- **TEST-1** (tier: unit) [covers: ITEM-1] file: `src-app/desktop/tauri/src/modules/office_bridge/tools.rs` — asserts: `tool_list()` advertises `run_office_js` whose `inputSchema` marks both `doc_full_name` and `script` required (string), `additionalProperties:false`.

- **TEST-3** (tier: integration) [covers: ITEM-2] file: `src-app/desktop/tauri/tests/office_bridge/settings_mcp_test.rs` — asserts: the live `tools/list` JSON-RPC response advertises the new 7-tool set (updated `EXPECTED_TOOLS`); `edit_document` no longer present, `run_office_js` present.
