# Human Feedback — scheduled-tasks

Living ledger (Phase 9 gate). Each human feedback item is recorded VERBATIM as it
arrives, with a status and — where it generalizes beyond this feature — a candidate
fleet-wide rule. Any `[status: open]` fails the Phase 9 gate.

- **FB-1** [status: resolved] — "it did NOT reuse existing page/drawer layouts (chat page, project page, etc.) -- match those, do not invent new ones" → rebuilt `ScheduledTaskFormDrawer` on the shared `FormField`/kit primitives, mirroring the `ProjectFormDrawer`/chat-drawer structure (labelled Field rows, not raw flex-gap), and the list page reuses the settings-page container layout; no bespoke layout remains [generalizable: yes — a new feature's page/drawer/settings surfaces MUST mirror the closest existing layout (chat page, project page, settings card/drawer); never invent a bespoke layout when an established one fits]
- **FB-2** [status: resolved] — "the Drawer asks the user to TYPE the ID of an assistant and the ID of a workflow to run -- users never see those IDs and never would; replace every raw-ID text input with a proper selection picker" → added `TaskTargetPickers.tsx`: `AssistantField`/`WorkflowField` are name-searchable Comboboxes and `ModelField` is a grouped Select, each populated from the picker stores' cached lists and bound to the form via `FormField`; zero raw-ID text inputs remain [generalizable: yes — never ask a user to type a raw entity ID (assistant/workflow/model/provider/etc.); any field whose value is an entity reference MUST be a selection picker (dropdown/searchable list) populated from that entity's list endpoint, showing human-readable names]
- **FB-3** [status: resolved] — "who would try to input timezone? we need to auto detect timezone from user client" → the timezone input was removed; the browser IANA zone (`Intl.DateTimeFormat().resolvedOptions().timeZone`) is auto-detected, submitted with the task, and rendered read-only as `schedule-timezone-note` (no editable control) [generalizable: yes — never ask the user to input a value the client already knows (timezone, locale, etc.); auto-detect it from the browser/environment and, if it must be visible, show it read-only — do not render it as an input field]
- **FB-4** [status: resolved] — "we need to allow user to select what days (multiple) in the week to run it" → `ScheduleBuilder` weekly mode is now a Mon–Sun multi-toggle emitting a comma-separated cron DOW field (`0 9 * * 1,3,5`); `humanizeCron` renders the multi-day summary [generalizable: yes — a recurring-schedule builder must support selecting MULTIPLE days of the week (multi-toggle), not a single day; when a field naturally accepts multiple values, offer multi-select rather than single-select]
- **FB-5** [status: resolved] — "the Inputs of workflow, it needs to based on workflow, not just the json. the UX is horrible" → the drawer parses the selected workflow's IR (`parseWorkflowIr`) and renders one typed `FormField` per declared input (with required-input validation); the raw-JSON textarea survives only as the last-resort fallback when a workflow exposes no declared inputs [generalizable: yes — collect structured input via a form generated from the TARGET's declared schema/inputs (one typed field per input), never a raw JSON blob; a JSON textarea for structured input is a UX failure and is only acceptable as a last-resort fallback when no schema is available]
- **FB-6** [status: resolved] — "when implement anything ,we need to check every item to see how user experience it and see how it relates to all current infastructure and see each of the infastructure if they have specific things that we need to pay attention to. add that to the human feedback" → authored `INFRA_INTEGRATION.md` doing both walks (per-item UX walk + per-subsystem infrastructure walk over chat pipeline, MCP tool-call/approval, permissions, notifications, sync, streaming, workflow runner, settings); the unattended tool-call approval gap it surfaced drove the safe-default + per-task allow-list design (DEC-17, ITEM-13/14/16/17) [generalizable: yes — for EVERY plan item, before/while implementing, do two mandatory walks: (1) a USER-EXPERIENCE walk (how does a real user actually encounter, trigger, and live with this item end-to-end), and (2) an INFRASTRUCTURE-INTEGRATION walk (enumerate every existing subsystem the item touches — chat pipeline, MCP tool-call + approval flow, permissions, notifications, sync, streaming, workflow runner, settings, etc. — and for EACH, check whether it has specific behaviors/constraints that must be handled, not assumed). Record the findings as an item×infrastructure analysis artifact. Applies fleet-wide; this is now a lifecycle sub-step of Phase 5.]
