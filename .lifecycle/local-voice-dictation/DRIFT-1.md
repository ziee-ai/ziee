# DRIFT-1 — implementation vs plan (phase 5)

Audit of the implemented `feat/local-voice-dictation` diff against PLAN.md /
DECISIONS.md after the full backend + frontend build. Each divergence is
reconciled below.

- **DRIFT-1.1** — verdict: resolved — Migration numbers shifted 132/133 → **133/134**: `00000000000132_add_openrouter_provider_type.sql` already exists on origin/main (missed in the phase-2 `ls | tail -3`). Renumbered `create_voice`→133 + `grant`→134; updated the code comments + all lifecycle artifacts. build.rs applies both; `Repos.voice` queries verify against the migrated build DB.
- **DRIFT-1.2** — verdict: impl-wins — No `VoiceInstance` **sync entity**; the managed-instance state is read live via `GET /api/voice/instance` (the frontend `VoiceInstance` store polls / refetches on action) instead of being pushed. Rationale: instance liveness is ephemeral runtime state (not a persisted admin record like settings/versions); a sync entity + fan-out would be overhead for data the admin only views on the instance card. `VoiceSettings` + `VoiceRuntimeVersion` (the durable records) DO sync. PLAN ITEM-18/22 wording ("instance-health block") is satisfied by the poll.
- **DRIFT-1.3** — verdict: impl-wins — The **model download** endpoint (`POST /api/voice/model/download`) is **synchronous** (returns the final `VoiceModelStatus`) with an indeterminate "Downloading…" UI state, rather than the SSE byte-progress bar DEC-15 implied. The **runtime-version** install DOES use the full detached-task SSE progress (`subscribeVersionDownloadEvents`), so the "full SSE progress" parity holds for the larger/again-fetchable engine binaries. A single ggml model fetch (≤ ~466 MB, one-shot, admin-initiated) is acceptable synchronous; an SSE model-progress variant is a documented follow-up (the `download_model_with_progress` callback seam is already in place).
- **DRIFT-1.4** — verdict: impl-wins — Instance **live-log SSE streaming** (`GET /voice/instance/logs/stream`, listed under ITEM-18) is **deferred**. Log capture + a ring buffer + a broadcast channel are wired in `deployment/local.rs` (`logs()`/`subscribe_logs()`), but the tail endpoint + a frontend log panel are not built. Marked `#[allow(dead_code)]` with DRIFT-1 pointers (not deleted — it's reserved, working plumbing). The instance card shows status/state/restart/stop, which covers the operational need; live logs are a debugging nicety.
- **DRIFT-1.5** — verdict: resolved — The pinned model sha256 table shipped as all-zero **placeholders**; DEC-6 requires real integrity pins. Fetched the real digests (the HF git-LFS `oid sha256` == the file's sha256) for `ggml-{tiny,base,base.en,small}.bin` and pinned them; downloads now hard-verify (mismatch → delete + fail). No longer a gap.
- **DRIFT-1.6** — verdict: impl-wins — ITEM-1 (`ziee-ai/whisper.cpp` **fork release workflow**) is **not built** — deferred by explicit user direction ("finish your implementation locally first, then we will deal with that later"). The fork repo exists; the CI + the naming-contract fixture (TEST-32) are a follow-up. The local runtime, its `MockReleaseServer`-driven tests, and the `stub-whisper-server` fixture do not need the real fork. The asset-naming contract the fork MUST satisfy is fixed in `engine/download.rs::archive_name` (`whisper-server-{platform}-{arch}-{backend}.{tar.gz|zip}`, with `{platform}`/`{arch}` = `gpu_detect::host_platform()/host_arch()` strings).
- **DRIFT-1.7** — verdict: impl-wins — `voice_runtime_settings.auto_start_timeout_secs` defaults to **60** (DEC-14 said 30). Whisper cold start includes a model load (base ≈ 142 MB) that can exceed 30 s on a slow disk; 60 s is the safer default (still admin-configurable 1..600). The `llm_runtime_settings` 30 s default assumes a resident engine binary + lazy model mmap; whisper-server loads the model at start.
- **DRIFT-1.8** — verdict: impl-wins — Desktop macOS mic support ships the `NSMicrophoneUsageDescription` **Info.plist** merge (the runtime-prompt requirement) but NOT a hardened-runtime `com.apple.security.device.audio-input` **entitlement** wired into the signing config. The usage-description is what makes `getUserMedia` prompt succeed in dev/unsigned builds; the entitlement is only needed under hardened-runtime signing and is part of the deferred desktop-release-signing config (out of local scope).
