- **FB-1** [status: resolved] — *"MUST FIX: in the logout handler, wrap the bump token_version + revoke_all_for_user in ONE DB transaction (both commit or neither), and publish Session only AFTER it commits — otherwise a bump-succeeds-but-revoke-fails window lets a held refresh token re-mint a valid access token through mint_session_tokens (which reads the new ver) and defeats logout; add a test that a forced revoke failure yields no re-mint / rolls back."* → Implemented as `refresh_tokens::end_session_atomically` (`pool.begin()` → bump → revoke → `commit()`, mirroring `claim_rotation_and_register` in the same file), with `sync_publish` moved strictly after the commit. `revoke_all_for_user`'s signature is left untouched so its other two callers stay out of scope. Covered by TEST-8, which installs a `BEFORE UPDATE` trigger on `refresh_tokens` that raises, and asserts the bump ROLLED BACK (`token_version` unchanged, access token still 200s, refresh token still rotates once the trigger is dropped). **This feedback also led directly to the branch's most serious find**: reasoning about the same invariant surfaced (a) the read-after-claim window in `refresh` (fixed by reading the epoch before the claim), and then (b) the blind audit's HIGH — a successor refresh token escaping a racing logout entirely, because under READ COMMITTED logout's revoke never scans an uncommitted successor. [generalizable: yes — when a fix introduces a new "both-or-neither" invariant across two tables, ask not only "are the writes atomic?" but "can a CONCURRENT transaction insert a row the revoking statement will never scan?" — READ COMMITTED makes uncommitted inserts invisible to an UPDATE's scan, so atomicity alone does not give you the invariant; you need the shared row lock.]
