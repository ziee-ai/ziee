# DECISIONS — chat-empty-completion-notice

All human/product inputs resolved up front so implementation runs nonstop.

### DEC-1: How is the empty completion surfaced to the user?
**Resolution:** An inline per-message `Alert` inside the assistant bubble, detected at render
time (so it also shows on reload). Not a conversation-level banner.
**Basis:** user

### DEC-2: What is in scope — just empty-completion, or also malformed tool-call deltas?
**Resolution:** Empty-completion only (turn ends with no visible content and no tool call). The
silently-discarded malformed tool-call-delta paths are deferred to a follow-up.
**Basis:** user

### DEC-3: Notice copy + tone?
**Resolution:** Copy = "The model returned an empty response and made no tool call." Tone =
`warning` (AlertTriangle ⚠ icon).
**Basis:** user — the approved AskUserQuestion preview showed the ⚠ icon and this exact copy.

### DEC-4: How is the new `data-testid` registered so `check:testid-registry` passes?
**Resolution:** The registry (`src-app/ui/src/components/ui/testIds.generated.ts`) is
auto-generated by a source scan (`scripts/gen-testid-registry.mjs`). Add the literal in
`ChatMessage.tsx`, then run `npm run gen:testid-registry`; `--check` (inside `npm run check`)
guards drift.
**Basis:** codebase — read `scripts/gen-testid-registry.mjs` (walks `src`, regex on `data-testid`).

### DEC-5: What backend signal accompanies the empty case?
**Resolution:** In the `_ =>` terminal arm, `tracing::warn!` + override the terminal
`finish_reason` to `"empty"` (a plain String on `SSEChatStreamCompleteData`). NOT an `Err`
(which would render as a hard conversation error banner) and NOT a new SSE variant.
**Basis:** convention/codebase — mirrors the `max_iterations` warn guard; the frontend
`complete` handler does not branch on finish_reason, so it is non-breaking; an `Err` path maps
to `SSEChatStreamEvent::Error` (`streaming.rs:809-819`), which is the wrong UX.

### DEC-6: What counts as a "user-visible answer" (so the flag stays false only when it should)?
**Resolution:** A `text` block with non-empty **trimmed** text, OR a `tool_use` / `tool_result`
/ `image` / `file_attachment` / `elicitation_request` block. `thinking` and empty/whitespace
`text` do NOT count.
**Basis:** codebase — the `MessageContentData` content-type vocabulary
(`chat/core/models/content.rs` + the text/mcp/file extensions).

### DEC-7: Is the notice dismissible?
**Resolution:** No (no `onClose`). It reflects durable message state, not a transient error.
**Basis:** convention — the kit `Alert` supports a non-dismissible form; dismiss is reserved
for transient/actionable errors (e.g. the conversation error banner).

### DEC-8: Which finalize loops set `produced_visible_content`?
**Resolution:** Both — the core content-block persist loop AND the extension-content persist
loop in `DeltaAccumulator::finalize`.
**Basis:** codebase — tool_use/file blocks are persisted via the extension loop
(`streaming.rs:~1419`), so a turn whose only visible output is an extension block must still
count as visible.

### DEC-9: Desktop UI mirror?
**Resolution:** None. The chat module is not duplicated under `src-app/desktop/ui`.
**Basis:** codebase — verified `src-app/desktop/ui/src/modules/chat/components/ChatMessage.tsx`
does not exist.
