# PLAN_AUDIT — chat-empty-completion-notice

Audit of PLAN.md against the actual codebase (read first-hand before writing code).

## Breakage risk
- **ITEM-3 (finish_reason override to `"empty"`)** — the terminal `Complete` SSE frame's
  `finish_reason` becomes `"empty"` instead of `"stop"` in the no-visible-content case.
  Checked consumers: the frontend `Chat.store.ts` `complete` handler (`:1502-1554`) does NOT
  branch on `finish_reason` values (only reads it into the event); no server code keys off
  `"stop"` specifically. `SSEChatStreamCompleteData.finish_reason` is a free `String`, so no
  enum/schema breaks. Persisted history is unaffected (finish_reason is not stored per
  message). Risk: LOW. Existing integration tests that assert `finish_reason:"stop"` on a
  NORMAL turn are unaffected (normal turns produce visible content → flag true → unchanged).
- **ITEM-1/2 (accumulator field + finalize flag)** — additive field with a default; the two
  finalize loops already iterate every persisted block, so setting the flag is O(1) per block,
  no new query, no transaction change. Risk: LOW.
- **ITEM-4/5 (frontend guard restructure)** — changing the `contents.length === 0` early
  return could regress the "files-only user message renders nothing" case. Mitigation: the new
  notice branch is gated on `!isUser && !isStreaming`; user/streaming messages keep the exact
  current `return null`. Risk: LOW, covered by TEST-4 + manual gallery.
- **ITEM-6 (testid registry)** — auto-generated by a source scan; regenerating is
  deterministic. Risk: NONE.

## Pattern conformance
- Backend mirrors the in-file `max_iterations` guard (`streaming.rs:211-226`) and the existing
  `_ =>` arm shape (`:666-689`) — same `ChatStreamChunk`/`tracing` idioms. `is_visible_answer`
  is a free function next to the other pure helpers (`accumulated_to_content_block:1117`,
  `group_assistant_blocks:1555`).
- Integration test mirrors `tests/chat/stub_chat_tier2_test.rs` (`StubChat`/`StubPlan`,
  `custom` provider, `helpers::{create_conversation, send_and_collect}`).
- Frontend `Alert` usage mirrors `ConversationPage.tsx:432/479` (`import { Alert } from
  '@/components/ui'`, `tone`, required `data-testid`). Tone = `warning` (AlertTriangle icon),
  matching the ⚠ surface approved by the requester.
- `hasVisibleAnswer` inspects `content_type` on `MessageContent` blocks — same discrimination
  the render run-loop already uses (`ChatMessage.tsx:75-82`).

## Migration collisions
- None. No DB migration added; `ls migrations/` unchanged. The fix adds no table/column —
  `produced_visible_content` is in-memory only, never persisted.

## OpenAPI regen
- **Not required.** `finish_reason` is an existing `String` field on `SSEChatStreamCompleteData`
  (`chat/core/types/streaming.rs:157-165`); emitting the value `"empty"` changes no schema. No
  new request/response type, no new enum. Therefore this diff is a backend+frontend behavior
  change with NO `openapi.json`/`types.ts` regeneration, and the frontend gate treats
  `ChatMessage.tsx` as the UI surface (e2e required — enumerated as TEST-4).
